Problem & positioning

The composition tool market is split: notation editors (Flat, Noteflight, MuseScore) have no AI, while AI sketchpads (Hookpad) have no real staff. A musician who plays trumpet or piano, wants to capture ideas fast, and needs an AI collaborator who speaks music theory — not just "generate something" — has no tool. Aria fills that exact gap.

What existing tools miss

  • Flat.io — best web notation UI, zero AI features
  • Hookpad — AI chord suggestions, no staff view, pop-biased
  • Noteflight — records audio but outputs a waveform, not notation
  • MuseScore — gold standard engraving, desktop only, no AI, not a sketching tool
  • AIVA / Suno — generate audio from prompts, no score, not editable

Aria's unique position

  • The only tool combining a real staff with an AI that understands what's on it
  • Mic → quantized, editable notation — no competitor does this in-browser
  • Natural language directing: "reharmonize this with a tritone sub" actually works
  • Built by and for someone who plays trumpet + piano — the AI speaks that vocabulary
  • Score branching: audition AI suggestions risk-free without touching your original

Core features — MVP

1. Score editor

Click-to-place notation on a rendered staff via VexFlow. This is the foundation everything else builds on.

  • Note and rest placement, ties, dots, accidentals
  • Multi-staff layout (e.g. trumpet + piano grand staff)
  • Time signature, key signature, tempo changes
  • Dynamics and articulations (mp, f, sfz, staccato, tenuto)
  • Undo / redo stack, keyboard shortcuts

2. Playback

Real instrument samples, not MIDI beeps. Built on Tone.js + SoundFont2 samples.

  • Per-staff instrument assignment (trumpet, piano, violin, etc.)
  • Solo / mute per staff
  • Loop region for practicing a section
  • Tempo slider, playhead scrubbing
  • SoundFont2 samples loaded from CDN (no backend)

3. Audio input (mic → notation)

Hum or play your instrument — it appears as editable notes on the staff.

  • Basic Pitch WASM runs entirely in the browser
  • Outputs MIDI pitch + timing; snapped to current time sig + quantization
  • Monophonic input: trumpet, voice, single-line instruments
  • Confidence threshold: only notes above threshold are placed
  • Review before commit: recorded notes preview before being written to staff

4. Transposition & instruments

Smart transposition, not mechanical semitone shifting.

  • Concert pitch → Bb trumpet → Eb alto → F horn toggle
  • Enharmonic spelling: Db vs. C# chosen by key context
  • Range warning system — idiomatic per instrument ("above the staff for Bb trumpet")
  • Reassign any staff to a different instrument, with range recheck

5. AI panel (Claude)

Select measures, describe what you want. Claude reads the score context and writes back to the staff.

  • Selection → MusicXML serialization → Claude API prompt
  • Supported commands: harmonize, reharmonize, add countermelody, extend, simplify, transpose style
  • Response parsed from MusicXML back into note events, previewed before commit
  • Chat history context — "make that even darker" understands the previous request
  • Explains its reasoning ("I used a borrowed iv chord here because...")

6. Score branching

The risk-free AI exploration model — borrowed from version control.

  • Fork current score into a named branch at any point
  • Audition AI suggestions in a branch without touching the original
  • Merge a branch back, or discard it — original is always intact
  • Branch list in sidebar with playback preview per branch

AI layer — how Claude is integrated

The prompt pipeline

User selects measures + types request
→
Selection serialized to compact MusicXML
→
Sent to Claude with full score context + instrument info
→
Claude returns modified MusicXML for that region
→
Parsed → previewed on staff → user commits or discards

System prompt design

  • Claude is given the full score context: key, time sig, tempo, all staves, instrument names
  • Instructed to return only valid MusicXML for the modified region — no prose, no backtick fences
  • Constraint injection: instrument ranges, voice-leading rules, style instructions from user preferences
  • Musician-level vocabulary expected: "tritone sub", "Neapolitan chord", "con sordino" are valid inputs

Parsing & error handling

  • Strip any markdown fences from response before XML parsing
  • Schema validate against MusicXML 3.1 spec before rendering
  • Graceful fallback: if parse fails, show raw text + "try again" rather than crashing
  • Range check after insertion: any out-of-range notes are flagged immediately

Risks & mitigations

Claude returns invalid or unparseable MusicXML

High

Strict system prompt with few-shot MusicXML examples. Schema validate every response. Graceful fallback: show raw text + "regenerate" button rather than crashing the editor.

Basic Pitch transcription quality on trumpet

High

Basic Pitch was trained on guitar/piano — trumpet's overtone series may cause pitch detection errors. Mitigate with confidence thresholding and a review-before-commit step. Consider fallback to Web Audio frequency estimation for brass.

VexFlow API complexity for full notation

Medium

VexFlow's API is verbose but well-documented. Scope MVP to essential notation symbols (notes, rests, ties, basic dynamics). Defer ornaments, extended techniques, and complex beaming to post-MVP.

Score state complexity as features compound

Medium

Define the full TypeScript score data model in Week 1 before writing any UI. Use Immer for immutable state updates in the reducer. Treat the score model as the single source of truth — VexFlow, Claude, and exports all derive from it.

SoundFont loading latency on first play

Low

Lazy-load SoundFonts per instrument on first assignment, not on page load. Show a loading indicator. Cache via Service Worker after first load.
